# Preventive Action Template

このテンプレートは、rollback 後の調査結果を再発防止アクションへ変えるためのものです。目的は、`調べた` で止めず、`次に何を直すか` を優先度つきで残すことです。

## 事象の要約

- 何が起きたか:
- いつ起きたか:
- 利用者影響:
- rollback で収束したか:

## いま分かっていること

- 事実:
- まだ仮説のもの:
- 再現できるか:

## 再リリース前に必須な対策

- 例 `readiness probe の閾値見直し`
- 例 `canary Pod の resource limit 修正`
- 例 `VirtualService の weight 確認手順を review checklist に追加`

## 今週中にやる対策

- 例 `istio-proxy ログ確認コマンドを runbook に追加`
- 例 `5xx 増加を検知する alert を追加`
- 例 `Hold 判断の観測時間を team ルール化`

## 後で進める恒久対策

- 例 `負荷試験を canary 前の標準手順にする`
- 例 `リリース判定の dashboard を team 共通化する`
- 例 `rollback 後の evidence 回収を CI から補助できるようにする`

## 担当と期限

- 再リリース前に必須な対策の担当:
- 今週中にやる対策の担当:
- 恒久対策の owner:
- 次回確認日:

## 完了条件

- 再リリース前に必須な項目が終わっている
- 再リリースの判断基準が前回より明確になっている
- 同じ失敗を次回は早く検知できる状態になっている
- [rerelease-readiness-checklist.md](rerelease-readiness-checklist.md) で再リリース可否を確認できる

## 書き方のコツ

- `原因っぽい` だけで対策を決めない
- 利用者影響を下げるものを先に置く
- `監視を強くする`, `設定を直す`, `運用を変える` を分ける
- 対策ごとに owner と期限を入れる

## このテンプレートで学ぶこと

- rollback 後は調査だけでなく改善計画まで必要なこと
- 実務では対策を優先度順に並べる必要があること
- 技術対策と運用対策の両方が再発防止に効くこと
