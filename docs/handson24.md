# Handson 24

## テーマ

Grafana での確認順序を学ぶ。

## 今回のゴール

- Grafana を `眺める` のではなく `順番を持って確認する` 習慣を作る
- 最初に見るべきパネルと後回しでよいパネルを分けられる
- 障害時や canary 中に確認順序が重要な理由を説明できる

## この回で先に押さえる用語

- Dashboard: 複数のグラフをまとめた画面
- Panel: 個別のグラフや表示領域
- Triage: 何から先に見るかを決める初動整理
- Critical Signal: 最優先で見るべき重要な指標
- Secondary Signal: 補助的に使う指標

## 対応ファイル

- [grafana-canary-checklist.md](grafana-canary-checklist.md)

## この回で実際にやること

1. [grafana-canary-checklist.md](grafana-canary-checklist.md) を読み、見る順番を整理する
2. `なぜその順番なのか` を自分の言葉で説明する
3. `最初の 2 分で見るもの` と `その後に深掘るもの` を分ける
4. 自分ならどこで Hold、どこで Rollback を考えるかを書き出す

## 完了条件

- dashboard を闇雲に開かず、確認順序を説明できる
- 最優先指標と補助指標を分けて説明できる
- `順番がない監視` がなぜ弱いか説明できる

## 実務で見る観点

- 影響の大きい 5xx や restart を先に見ているか
- CPU や Memory のような補助指標に引っ張られすぎていないか
- 時間がない状況で `まずこれを見る` が決まっているか

## よくある失敗

この回の失敗は、画面を順に開くだけで、どの順番で深掘るかの基準を持たないと起きやすいです。箇条書きは別々に見えても、背景には `広く見てから狭く見る` 観察順序の不足があります。

- 目についた panel から見始めて全体像を失う
- CPU や Memory に時間を使いすぎて 5xx 確認が遅れる
- dashboard を切り替え続けて、結局判断基準が曖昧になる

## 学ぶポイント

- Grafana では `何を見るか` だけでなく `どの順番で見るか` が重要である
- 初動では利用者影響に近い指標を先に見る方が強い
- 監視順序が決まっていると、障害時の思考負荷を下げられる

## 学ぶポイントの解説

Grafana の使い方が弱いチームでは、dashboard はあるのに初動が遅くなりがちです。理由は、情報が多すぎて何から見ればよいか決まっていないからです。実務では、時間のない中で `まず異常かどうか`, `利用者影響があるか`, `変更起因か` を早く判断する必要があります。

そのため、最初に 5xx、restart、latency のような重要シグナルを見て、その後に CPU や Memory で原因を補強する順番が有効です。順番があるだけで、Grafana は `きれいな画面` から `判断の道具` へ変わります。

- `今の順番だと見落としやすいもの` を 1 つ挙げる
