# Handson 1

## テーマ

この学習で何を作るのかを理解する。

## 今回のゴール

- 最終構成を説明できる
- なぜその構成が Kubernetes 学習に向いているか説明できる
- 何をどの順で学ぶのか把握できる

## 先に押さえる用語

この回では、次の用語が分かっていると読みやすくなります。

- Cluster: Kubernetes 全体の実行環境
- Node: Pod が動くマシン
- Pod: コンテナを実行する最小単位
- Deployment: stateless なアプリを管理する仕組み
- StatefulSet: DB のような stateful workload を管理する仕組み
- Service: Pod への安定した接続先
- Ingress: HTTP/HTTPS の入口
- GitOps: Git を正として変更を管理する運用

用語に不安があれば、先に [glossary.md](glossary.md) を読んでから進めてください。

## 作るもの

kind 上に次の小さな業務システムを作ります。

- PostgreSQL
- user-service
- item-service
- web-app
- Grafana
- Argo CD
- Ridgepole によるスキーマ管理

## 想定アーキテクチャ

- 入口: Ingress NGINX
- フロント: web-app
- API: user-service, item-service
- DB: PostgreSQL
- 監視: Prometheus, Grafana
- GitOps: Argo CD
- スキーマ変更: Ridgepole Job または CI

## なぜこの構成にするのか

目的:

- stateless と stateful の違いを同じ環境で比較する
- 本番運用で必要になる監視と GitOps まで一気通貫で学ぶ

実務上のメリット:

- 単にアプリを動かすだけでなく、運用の論点まで触れられる
- 小さいが現実的なシステム境界を学べる

## この回で確認すること

- PostgreSQL は stateful workload だと理解できるか
- API と Web は stateless として扱いやすいと理解できるか
- Grafana 単体ではなく Prometheus と組み合わせる理由を説明できるか
- Argo CD を使うと Git が正になる意味を説明できるか

## 学ぶポイントの解説

この回で最も大事なのは、Kubernetes を単なるアプリ実行基盤として見ないことです。実務では、アプリを起動できるだけでは価値が足りません。障害時にどう戻すか、変更をどう安全に入れるか、状態を持つコンポーネントをどう扱うかまで含めて設計します。このシリーズが PostgreSQL、監視、GitOps まで含んでいるのはそのためです。

まず stateless と stateful の違いを意識してください。Web や API は Pod を入れ替えても本質的な状態を持たないように作りやすい一方、DB はデータそのものが価値なので、Pod を消してもデータを失わない設計が必要です。この違いを最初に理解しておくと、以降の StatefulSet や PVC の話が自然につながります。

監視と GitOps を最初から視野に入れる理由も重要です。監視は障害を見つけるだけでなく、システムの変化を定量的に把握するために必要です。GitOps は便利なデプロイ手法ではなく、変更履歴を残し、環境差分を抑え、再現性を上げるための運用設計です。実務で強い人は、最初から運用まで含めて全体像を見ています。

## この回の宿題

次の問いに自分の言葉で答えてください。

- なぜ DB と API を同じ Deployment にしないのか
- なぜ最初から監視と GitOps まで含めて学ぶ価値があるのか

## 実務で見る観点

- システムをアプリ、データ、監視、デプロイ運用の 4 層で分けて見られるか
- 今後の変更で、どこが頻繁に変わり、どこが慎重に変わるべきかを言えるか

## よくある失敗

- Kubernetes を「コンテナを並べる場所」とだけ捉えて運用設計を後回しにする
- DB、監視、GitOps を別テーマだと思い、アプリ本体と切り離して考えてしまう

## 詰まったときの確認コマンド

```bash
kubectl get nodes
kubectl get ns
kubectl get all -A
```

## 障害シナリオと復旧の考え方

想定シナリオ:

- アプリは起動しているが、障害時にどこを見ればよいか誰も分からない
- DB、API、監視、GitOps の責務が曖昧で、変更時に影響範囲を説明できない

復旧の考え方:

- まずシステムを層に分けて観察する
- どの層で問題が起きているかを切り分けてから詳細調査に進む

## レビュー観点と運用判断ポイント

- この構成は将来の変更単位を小さく保てるか
- 監視とデプロイ運用が後付けではなく最初から前提に入っているか
- stateful と stateless を混同した設計になっていないか

## 模擬インシデント演習

演習内容:

- チームメンバーが「まずアプリだけ動けばよい」と言って監視と GitOps を後回しにしようとしている状況を想定する

考えること:

- 後回しにした場合、どのタイミングで何が運用負債になるか
- 今の段階で最低限入れておくべき運用要素は何か

## レビューコメント例

- 「アプリ構成は分かりますが、障害時にどこを見るかの導線がまだ弱いです。監視と変更管理を最初から前提にした説明を追加した方が、運用設計として破綻しにくいです。」
- 「DB と API の責務が近く見える記述があるので、stateful と stateless の違いを最初に明示した方が後続の handson につながりやすいです。」

## 宿題の考え方

DB と API を同じ Deployment にしない理由を考えるときは、更新頻度、障害の影響範囲、スケールの仕方が同じかどうかを軸に考えてください。API だけ更新したい、API だけ増やしたい、DB だけ慎重に扱いたい、という要求が現場では常に出ます。責務の違うものを同じ単位で動かすと、その柔軟性が失われます。

監視と GitOps を早めに学ぶ価値については、開発の終盤で慌てて足す対象ではなく、最初から設計に組み込む対象だと考えると整理しやすいです。動くものを作った後で運用を足すより、運用前提で作る方が事故は減ります。この視点は、実務に入ったときに大きな差になります。
