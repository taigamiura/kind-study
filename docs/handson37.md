# Handson 37

## テーマ

Postmortem を書いて運用改善へつなげる。

## 今回のゴール

- 障害の学びを文章で残せる
- `何が起きたか`, `なぜそう判断したか`, `何を改善するか` を分けて書ける
- Postmortem を犯人探しでなく改善設計資料として扱える

## この回で先に押さえる用語

- Postmortem: 障害後のふりかえり文書
- Timeline: 時系列
- Root Cause: 根本要因
- Contributing Factor: 悪化に寄与した要因
- Action Item: 改善タスク

## 対応ファイル

- [postmortem-template.md](postmortem-template.md)

## この回で実際にやること

1. [postmortem-template.md](postmortem-template.md) を読み、`概要`, `時系列`, `改善` の 3 つに分ける
2. handson22 か handson36 の障害例を使って 1 つ postmortem を書く
3. preventive action と rerelease 条件へ再接続する
4. `うまくいったこと` と `改善が必要なこと` を両方書く

## 完了条件

- 障害の学びを文章で残せる
- `事実`, `判断`, `改善` を分けて書ける
- postmortem を preventive action へつなげられる

## 実務で見る観点

- 障害の背景要因まで掘れているか
- うまくいった対応も残して再利用できるか
- 個人ではなく仕組みの改善へつながっているか

## よくある失敗

- 時系列が曖昧で何が先だったか分からない
- 原因探しが個人批判になってしまう
- 復旧で終わった気になり、改善へつながらない

## 学ぶポイント

- postmortem は責任追及ではなく学習のための文書である
- 時系列と判断理由を残すと、次回の初動が速くなる
- 良い postmortem は preventive action と rerelease 判断へつながる

## 学ぶポイントの解説

障害対応が実務レベルかどうかを分ける最後の差は、復旧後に学べるかどうかです。postmortem を書かないと、復旧はその場しのぎで終わりやすく、同じ障害を繰り返します。

一方で、良い postmortem はチームの資産になります。何が起きたか、どこで判断したか、どの改善が必要かが整理されていれば、次回の release も障害対応も確実に強くなります。この回は、その `学びを残す技術` を身につけるための最終章です。

## この回の宿題

- 1 つ障害例について、`うまくいったこと` を 3 つ書く
- `個人の注意` で終わる改善案を、仕組みの改善へ言い換える

この handson37 までで、構築、配備、監視、セキュリティ、段階リリース、復旧、更新運用、再発防止、再リリース判断、障害ふりかえりまで一続きで学べる構成になります。
