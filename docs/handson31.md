# Handson 31

## テーマ

再リリース可否の判定を学ぶ。

## 今回のゴール

- `修正した` と `再度出してよい` は別の判断だと理解する
- 再リリース前に確認すべき技術条件と運用条件を説明できる
- 同じ失敗をそのまま繰り返さない gate の考え方を身につける

## この回で先に押さえる用語

- Re-release: 一度 rollback した変更を再度出すこと
- Readiness: 再度出してよい準備状態
- Gate: 次へ進む前に通る確認ポイント
- Validation: 直した内容の検証
- Residual Risk: まだ残っているリスク

## 対応ファイル

- [rerelease-readiness-checklist.md](rerelease-readiness-checklist.md)

## この回で実際にやること

1. [rerelease-readiness-checklist.md](rerelease-readiness-checklist.md) を読み、再度 canary を出してよい条件を整理する
2. `直した` と `再リリースしてよい` の違いを言葉にする
3. 前回の rollback 原因が本当に潰れたかをどう確認するか書く
4. 再リリース時の rollback 条件と観測時間を前回からどう改善するか考える

## 完了条件

- 再リリース判定を説明できる
- 技術修正だけでなく運用条件も見直す必要性を説明できる
- `まだ出すべきでない状態` を具体例で言える

## 実務で見る観点

- 前回の失敗条件が本当に解消しているか
- rollback 条件や観測時間が前回より改善されているか
- チーム共有文や判断基準も更新されているか

## よくある失敗

- 修正コミットが入っただけで再リリースしてしまう
- 監視条件や観測時間を前回のまま流用する
- `原因はたぶんこれ` の段階で再度出してしまう

## 学ぶポイント

- 再リリースは `修正済み` ではなく `再度出してよい条件を満たした` で判断する
- 技術修正だけでなく監視、共有、rollback 条件も更新が必要である
- re-release gate があると同じ障害の再発を減らしやすい

## 学ぶポイントの解説

rollback 後に最も危険なのは、焦って再度出してしまうことです。特に開発エンジニアがインフラ運用に不慣れな場合、`直したはず` を `出してよい` と混同しやすくなります。しかし実務では、その間に確認すべき gate がいくつもあります。

前回の原因仮説がどこまで検証されたか、rollback 条件が明文化されたか、観測時間は十分か、共有文まで更新したか。こうした条件がそろって初めて、再リリースはチームとして defensible になります。

## この回の宿題

- `再リリースしてよい条件` を 5 項目にまとめる
- `まだ出してはいけない状態` を 3 パターン書く
