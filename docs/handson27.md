# Handson 27

## テーマ

チームへの共有文の作り方を学ぶ。

## 今回のゴール

- 技術判断をチームが分かる文へ変換できる
- `何をしているか`, `何を見ているか`, `次に何をするか` を短く共有できる
- 技術的には正しいが伝わらない文章を避けられる

## この回で先に押さえる用語

- Status Update: 進行状況の共有
- Audience: その文章を読む相手
- Escalation: 状況悪化時に共有を強めること
- Mitigation: 影響抑制の行動
- Closure: 監視や対応を閉じること

## 対応ファイル

- [release-communication-template.md](release-communication-template.md)

## この回で実際にやること

1. [release-communication-template.md](release-communication-template.md) を読み、開始、観測中、rollback、promote 完了の違いを整理する
2. それぞれ 1 文ずつ書く
3. `技術者向けの文` と `関係者向けの文` の違いを考える
4. 原因断定しないまま共有する書き方を練習する

## この回だけで押さえる整理

共有文の強さは、技術的に詳しいことではなく、状況と影響と次の更新予定が短く伝わることです。この回では、技術者でない読み手でも危険度と今後の動きが分かる文章へ落とせるかがポイントです。

## 確認するとしたらどこを見るか

- 共有文では `状況`, `影響`, `今見ている事実`, `次の更新予定` が短く入っているかを見る
- 読み手が技術者でなくても、いま危険か安全かが分かる表現になっているか確認する

## 完了条件

- 技術判断を短く共有できる
- 共有文に `現在`, `根拠`, `次の行動` が入っている
- Rollback 時に `影響抑制を優先する` と明記できる

## 実務で見る観点

- 共有文が短いだけでなく、判断材料が含まれているか
- 読み手が技術者でなくても、現在のリスクが分かるか
- 状況悪化時の共有頻度や強度を上げられるか

## よくある失敗

この回の失敗は、技術的には分かっていても、共有文として相手に伝わる形へ落とせないと起きやすいです。箇条書きは別々に見えても、背景には `誰に何を伝える文か` の整理不足があります。

- 技術用語が多すぎて、何が起きているか伝わらない
- `様子見します` だけで、何分見るのか書かない
- rollback の理由が曖昧で、必要性が共有されない

## 学ぶポイント

- 技術判断は共有されて初めてチーム運用になる
- 良い共有文は短くても、根拠と次の行動が入っている
- 原因断定前でも、状況と判断は共有できる

## 学ぶポイントの解説

現場で強いエンジニアは、技術的に正しいだけでなく、その判断を周囲が使える形で伝えられます。特に canary や障害対応では、`いま 10% で見ている`, `再起動増加があるので rollback する`, `10 分後に再判断する` といった共有があるだけで、チームの認識ずれが減ります。

共有文は長ければ良いわけではありません。必要なのは、現在地、根拠、次の行動の 3 つです。この型を身につけると、リリースも障害対応もかなり安定します。

## この回の宿題

- Promote、Rollback、Hold の 3 パターンで共有文を 1 つずつ書く
- `技術的には正しいが伝わりにくい文` を 1 つ作り、どう直すか考える
