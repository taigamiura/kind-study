# Handson 35

## テーマ

SLO / Alert テンプレートの作り方を学ぶ。

## 今回のゴール

- `見る値` と `起こす Alert` を分けて考えられる
- 利用者影響に近い SLO を置く発想を持てる
- Alert が多いほど良いわけではない理由を説明できる

## この回で先に押さえる用語

- SLI: 測定する指標
- SLO: 目標値
- Error Budget: 許容できる失敗の余地
- Alert Fatigue: 通知疲れ
- Pager: 即時対応が必要な通知経路

## 対応ファイル

- [slo-alert-template.md](slo-alert-template.md)

## この回で実際にやること

1. [slo-alert-template.md](slo-alert-template.md) を読み、SLI / SLO / Alert の関係を整理する
2. user-service 向けに 1 つ SLO を書く
3. Dashboard で見る値と Pager を鳴らす条件を分ける
4. `通知する価値がない Alert` の例を 2 つ挙げる

## 完了条件

- `見る値` と `起こす Alert` を分けて説明できる
- SLO が利用者影響に近い理由を説明できる
- Noise を減らす考え方を説明できる

## 実務で見る観点

- SLO がチームの判断基準として使えるか
- Alert が人を起こす価値のある条件に絞れているか
- Error Budget を使って変更速度を調整できるか

## よくある失敗

この回の失敗は、テンプレートを埋めること自体が目的になると起きやすいです。箇条書きは別々に見えても、背景には `この SLO/Alert を誰がどの判断に使うか` を先に定めていないことがあります。

- CPU 高騰や一時的な spike を全部 Alert にする
- 利用者影響と遠い指標だけで運用する
- SLO の根拠が共有されていない

## 学ぶポイント

- 良い監視は大量の通知ではなく、良い判断基準を生む
- SLO は利用者影響に近い指標から始めると実務に乗りやすい
- Alert は人の集中力を使うので、価値があるものに絞るべきである

## 学ぶポイントの解説

開発エンジニアが監視設計でつまずくポイントは、見えるものを全部通知したくなることです。しかし実務では、人が本当に起きるべき条件を絞らないと運用が壊れます。ここで SLO が役立ちます。

SLO を先に置くと、どの失敗が許容範囲を超えたのか、どこで変更速度を落とすべきかが見えやすくなります。この回では、監視を `観察` から `運用基準` へ変える考え方を固めます。

- user-service に対して 1 つ SLO、1 つ即時対応 Alert、1 つ営業時間内 Alert を作る
- `通知しない方がよい指標` を 2 つ挙げ、その理由を書く
