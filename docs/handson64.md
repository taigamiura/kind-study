## この回で実際にやること

1. [cluster-rebuild-drill.md](cluster-rebuild-drill.md) を読む
2. `ゼロから cluster を戻す` 前提で必要な情報を列挙する
3. handson10、19、32、33、44、56 を使って復元手順を書く
4. 復元できないもの、手順化されていないものを洗い出す

## この回だけで押さえる整理

再構築訓練の目的は、`壊れたものをその場で直す力` ではなく、`前提を失っても再現できる力` を確認することです。クラスタを 1 回消しても戻せるなら、Git、manifest、runbook、backup、Secret 管理が本当に機能していると証明できます。

復旧の材料は少なくとも 5 系統に分かれます。

- cluster を作る材料: kind / managed cluster / Terraform など
- workload を戻す材料: manifests、Helm values、GitOps 定義
- data を戻す材料: backup、restore 手順、RPO/RTO
- 接続を戻す材料: DNS、Ingress、証明書、外部依存設定
- 運用を戻す材料: runbook、権限、監視、当番導線

どれか 1 つでも欠けると、`クラスタは立ったが業務システムとして戻らない` 状態になります。

## 確認するとしたらどこを見るか

- 再構築手順では Git、manifest、Secret、backup、DNS 切替のどこが手順化されているかを見る
- `戻せるはず` ではなく、ゼロから再構築して本当に戻るか確認できる形かを見る

## 完了条件

- cluster 再構築の大まかな流れを説明できる
- 復旧に必要な材料を列挙できる
- 現状の復旧手順の穴を指摘できる

## 実務で見る観点

- 復旧手順が特定個人の頭の中に閉じていないか
- Secret や外部接続情報が repo 外に散らばっていないか
- `戻るまでの時間` と `失ってよいデータ量` を説明できるか

## よくある失敗

- Git があるから復旧できると思い込む
- cluster だけ再作成して data や DNS の戻し方を持っていない
- 復旧訓練をせず、runbook の正しさを信じ込む

## 学ぶポイントの解説

実務で強いチームは、障害時の improvisation ではなく、再構築可能性で勝負しています。再構築訓練をすると、普段は見えない `Secret の出し方が分からない`, `DNS 切替手順がない`, `restore の整合確認が弱い` といった穴が露出します。

また、再構築は技術作業だけではありません。どの順で人が動くか、誰が承認するか、どこで切替判断するかまで含めて初めて運用になります。この視点まで持てると、DR の理解が一段深まります。

## 学ぶポイント

- 復旧力は runbook と訓練でしか育たない
- Git と backup があっても手順が弱ければ戻せない
- 再構築訓練は運用成熟度を測る実務的な方法である
