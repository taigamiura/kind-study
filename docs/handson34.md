# Handson 34

## テーマ

Secret / Certificate rotation の確認を学ぶ。

## 今回のゴール

- Secret / Certificate の更新を安全な day-2 運用として理解する
- 更新前、更新直後、反映後で何を確認するか説明できる
- `値を変える` と `安全に反映する` の違いを理解する

## この回で先に押さえる用語

- Rotation: 値の更新
- Renewal: 証明書更新
- Rollout Restart: 更新値を Pod へ反映する再起動
- Blast Radius: 変更の影響範囲
- Expiry: 期限切れ

## 対応ファイル

- [secret-rotation-checklist.md](secret-rotation-checklist.md)

## この回で実際にやること

1. [secret-rotation-checklist.md](secret-rotation-checklist.md) を読み、更新前後の確認を整理する
2. 更新時の見落としポイントを 3 つ書く
3. どの workload に影響するかを先に洗い出す理由を説明する
4. 更新後にどの指標を見るべきか整理する

## 確認するとしたらどこを見るか

- rotation では更新前の依存先、更新直後の resource 状態、反映後の rollout 完了を分けて見る
- 値を更新しただけでなく、実際に Pod が新しい値を使っているか確認できる形になっているかを見る

## 完了条件

- rotation の確認順序を説明できる
- 影響範囲を先に洗い出す必要性を説明できる
- 証明書では Ready だけでなく期限を見る理由を説明できる

## 実務で見る観点

- Secret の owner と更新窓口が明確か
- 期限切れ前に更新できる運用になっているか
- 更新後の Pod 状態と error 監視までセットで見ているか

## よくある失敗

この回の失敗は、rotation を値の更新だけで終わらせると起きやすいです。箇条書きは別々に見えても、背景には `影響確認まで含めて運用` する意識の不足があります。

- Secret を変えても workload 反映を忘れる
- 証明書の期限管理が手作業に依存する
- 影響範囲を把握せず一斉変更してしまう

## 学ぶポイント

- Secret / Certificate 運用は `更新`, `反映`, `確認` の 3 段階で考える
- 期限切れは設定ミスというより運用ミスで起きやすい
- day-2 運用では変更範囲の把握が事故防止に効く

## 学ぶポイントの解説

Secret や証明書は、導入した瞬間より更新時に事故が起きやすいです。特に開発エンジニアがインフラを兼務する場合、`値を変えたから終わり` と考えやすく、再起動や影響確認が抜けやすくなります。

この回では、更新の前後で何を見るべきかを運用手順として理解します。ここが固まると、認証情報や証明書の運用に対する心理的ハードルがかなり下がります。

## この回の宿題

- Secret 更新時に確認を忘れると危険な項目を 5 つ挙げる
- 証明書更新が遅れたときに起こりうる障害を 2 つ書く
