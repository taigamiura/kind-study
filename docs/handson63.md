# Handson 63

## テーマ

production readiness review を実践する。

## 今回のゴール

- 本番投入前に何を審査すべきかを体系的に説明できる
- 監視、復旧、権限、セキュリティ、変更管理、運用体制を一体で確認できる
- `動く` と `本番に出せる` の差を説明できる

## 対応ファイル

- [production-readiness-review.md](production-readiness-review.md)

## この回で実際にやること

1. [production-readiness-review.md](production-readiness-review.md) の全項目を確認する
2. 未充足項目を一覧化する
3. `本番投入可`, `条件付き可`, `不可` の 3 段階で判定する
4. 判定理由と未充足項目の改善順序を書く

## この回だけで押さえる整理

production readiness review は、`デモが通るか` を見る場ではありません。`本番で事故が起きたときに戻せるか、監視できるか、権限と変更統制があるか` を審査する場です。技術的な動作確認より、運用可能性の審査に近いと考えると整理しやすくなります。

判定は 3 段階で考えます。

- 本番投入可: 主要な欠落がなく、残課題は投入後改善でよい
- 条件付き可: 動かせるが、期限付きで埋めるべき不足が残る
- 不可: rollback plan、監視、権限、復旧、運用体制などの必須条件が欠けている

この回のポイントは、`何がないと不可か` を説明できることです。すべて完璧でなくてもよい一方で、欠けてはいけない最低線があります。

## 確認するとしたらどこを見るか

- readiness review では監視、復旧、権限、変更管理、運用体制に抜けがないかを見る
- 技術的に動作していても、当番体制や rollback plan がない状態を見落としていないか確認する

## 完了条件

- readiness review の観点を説明できる
- 判定と根拠を文章で残せる
- 本番投入前の不足を改善タスクへ落とせる

## 実務で見る観点

- 技術審査と運用審査を分けず、一体で見ているか
- 条件付き可にした項目が本当に期限管理されるか
- `動くから出す` 圧力に流されず、不可判定を言える基準があるか

## よくある失敗

- 本番投入の判定を雰囲気で決める
- 監視や rollback plan が弱いのに、機能が動くので通してしまう
- 未充足項目を列挙するだけで、Go/No-Go 判定に反映しない

## 学ぶポイントの解説

本番投入前に必要なのは、技術力より `止める判断を言語化する力` です。機能開発が進んでいるほど、投入を遅らせる判断は言いにくくなります。しかし readiness review は、まさにそこで最後の防波堤になります。

また、`条件付き可` を乱用すると review は形骸化します。条件付きで通すなら、何をいつまでに誰が埋めるかを明示しなければ意味がありません。この厳しさまで含めて理解できると、実務で使える review になります。

## 学ぶポイント

- readiness review は技術レビューより広い
- 本番投入可否は単一の観点で決めない
- 未充足事項を言語化できるほど実務に強い

## この回の宿題

- 自分の想定サービスに対する readiness review 結果を書く
- `不可` と判断する基準を明確に書く
