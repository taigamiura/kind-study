# Handson 23

## テーマ

リリース中に見るメトリクスの意味を学ぶ。

## 今回のゴール

- restart、CPU、Memory、5xx を `ただの数字` ではなく `判断材料` として読める
- どの変化を Promote、Hold、Rollback の根拠にするか説明できる
- `メトリクスを見る` と `メトリクスで判断する` の違いを理解する

## この回で先に押さえる用語

- Restart: Pod や container の再起動回数
- 5xx: サーバー側で失敗した HTTP 応答
- Hold: すぐ Promote も Rollback もせず追加観測する判断
- Baseline: 比較対象となる通常時の値
- Noise: 判断に使いにくい揺らぎや雑音

用語に迷ったら [glossary.md](glossary.md) の Metrics、Error Rate、Latency、SLO を見直してください。

## 対応ファイル

- [release-metrics.md](release-metrics.md)

## この回で実際にやること

1. [release-metrics.md](release-metrics.md) を読み、restart、CPU、Memory、5xx がそれぞれ何を示すか整理する
2. `すぐ rollback すべき変化`, `様子を見る変化`, `許容できる変化` を 1 つずつ書く
3. stable と canary の比較で、どの値を baseline にすべきか考える
4. 自分の canary 判断基準を 3 行程度で書く

## この回だけで押さえる整理

この回で重要なのは、メトリクスを `数字` として見るのでなく `判断材料` として見ることです。特に canary では絶対値より stable との差分、瞬間値より継続時間、CPU や Memory より利用者影響に近い 5xx や latency を優先して読むと判断がぶれにくくなります。

## 確認するとしたらどこを見るか

- メトリクス画面では stable と canary の差分、瞬間値ではなく継続時間、5xx や restart が片側だけ悪化していないかを見る
- CPU や Memory は単独でなく、利用者影響に近い latency や 5xx と一緒に見て判断する

## 完了条件

- 観測値を `promote / hold / rollback` に結び付けて説明できる
- `数字が高い` ではなく `なぜ危険なのか` を言葉で説明できる
- baseline がないまま判断すると危険な理由を説明できる

## 実務で見る観点

- 数値だけでなく、stable と canary の差分として見ているか
- restart や 5xx の一時的な揺らぎと、継続的な悪化を分けて見ているか
- CPU が上がっていても利用者影響がないケースを切り分けられるか

## よくある失敗

この回の失敗は、単発の数値や一瞬の変化だけを見て、比較対象や観測時間を持たないと起きやすいです。箇条書きは別々に見えても、背景には `何を根拠に進めるか / 止めるか` が曖昧なことがあります。

- CPU が上がっただけで即 rollback してしまう
- 5xx が一瞬出ただけで全体傾向を見ずに判断する
- canary だけを見るのではなく stable との比較を忘れる
- `普段このくらい` が分からないまま良し悪しを決める

## 学ぶポイント

- メトリクスは `見る` ためではなく `判断する` ために使う
- canary では絶対値より stable との差分が重要なことが多い
- 利用者影響に近い 5xx や latency を優先して見る方が実務的である

## 学ぶポイントの解説

リリース中の監視でありがちな失敗は、数字を集めること自体が目的になることです。実務で重要なのは、`この変化なら止める`, `この変化ならまだ見る`, `この変化なら進める` を事前に言えることです。そのため、メトリクスは情報ではなく判断基準として扱います。

例えば CPU 使用率は上がっても、レスポンスや 5xx に影響がなければ即 rollback の根拠にはならないことがあります。一方で 5xx が継続して増える、restart が canary 側だけで増える、latency が急に悪化する、といった変化は利用者影響に直結しやすいため優先度が高いです。

また、数値は単体で見るより比較で見る方が強いです。stable と canary を比べて canary 側だけが崩れているなら、変更起因の可能性が高まります。この `比較して読む` 発想は、以後の release 判断全体の土台になります。

## この回の宿題

- `即 rollback`, `Hold`, `Promote` の 3 パターンで、それぞれ判断に使う指標を 1 つずつ決める
- user-service の canary で最も先に見るべき値を 2 つ選び、その理由を書く
