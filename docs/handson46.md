# Handson 46

## テーマ

変更管理、緊急変更、運用統制を学ぶ。

## 今回のゴール

- GitOps があっても変更管理が必要な理由を説明できる
- 通常変更と緊急変更を同じ扱いにしない発想を持てる
- `誰が承認し, 何を記録し, どこで例外にするか` を整理できる

## この回で先に押さえる用語

- Change Management: 変更管理
- Emergency Change: 緊急変更
- Approval: 承認
- Audit Trail: 変更の追跡記録
- Freeze Window: 変更凍結期間

## 対応ファイル

- [change-management-guide.md](change-management-guide.md)

## この回で実際にやること

1. [change-management-guide.md](change-management-guide.md) を読み、通常変更と緊急変更の差を整理する
2. `誰が承認するか`, `何を記録するか`, `いつ例外を許すか` を書き出す
3. GitOps と緊急手動操作の関係を説明する
4. freeze window が必要な場面を考える

## 完了条件

- 変更管理が必要な理由を説明できる
- 緊急変更でも最低限残すべき記録を説明できる
- GitOps と change management の役割差を説明できる

## 実務で見る観点

- 本番変更の承認フローが曖昧でないか
- 緊急変更後に Git へ戻す運用があるか
- freeze window や高リスク時間帯の考え方があるか

## よくある失敗

この回の失敗は、変更管理を手続きだけの話として捉えると起きやすいです。箇条書きは別々に見えても、背景には `なぜ統制が必要か` と `どこまで例外を許すか` の基準不足があります。

- GitOps があるから統制は不要だと考える
- 緊急変更を例外のまま放置して drift を生む
- 誰が決めた変更か追えない

## 学ぶポイント

- 実務クラスタでは技術だけでなく変更統制が必要である
- 緊急変更は早さだけでなく記録の質が重要である
- change management は安全性と速度のバランスを取る仕組みである

## 学ぶポイントの解説

Kubernetes を実務で運用すると、技術的に正しい変更でも `いつ`, `誰が`, `どう承認して`, `どこへ残すか` が問われます。GitOps はその一部を助けますが、緊急変更や例外運用まで自動で整理してくれるわけではありません。

この回を通じて、クラスタ運用は技術だけでなく統制と記録も含むことを理解できるようになります。ここまで来ると、かなり実務に近い視点になります。
